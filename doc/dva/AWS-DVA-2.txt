AWS Step Functions overview
- [Instructor] In this lecture we'll talk about AWS Step Functions. Step Functions lets you orchestrate the components of a microservices application or any processes into a serverless workflow. A workflow is a series of steps with the output of one step acting as input into the next step. Your workflow defines the process of how your application accomplishes its goal from start to end. In Step Functions, we refer to the workflow as a State Machines, and each step in a State Machine is called a state. What's nice about Step Functions is that you can review your State Machine visually with its graphical user interface. This is a truly great feature as it can already serve as your blueprint, so you would have a big picture of what you're trying to do when designing a workflow. Having a visual workflow will help you track where you are, your inputs and outputs for each state, as you add more steps in your workflow or as you update it. A State Machine is defined using the Amazon States language or ASL, which has a similar syntax to adjacent document. If you don't want to write in JSON Step Functions, also provide a drag-and-drop interface, which further simplifies how you build your State Machine. There are eight state types that you can choose from when building a State Machine. These are the Task, Choice, Parallel, Wait, Fail, Succeed, Pass, and the Map state. So just a quick rundown on what they are. The first one, which is the core of your State Machine, is the task state. The Task State represents a single unit of work in your State Machine, application code that needs to be executed by your Lambda function, is done by the Task state. A task state can only point to a single lambda function, an easiest task or what have you. When you start the execution of a State Machine, the Step Functions will invoke Lambda function directly from a task state. Next is the choice state. The choice state adds a branching logic to your workflow, and you can think of it as an if then else statement where there is more than one possible outcome that can occur when evaluating a parameter against a condition that you set. Then we have the parallel state. The parallel state allows you to perform two branches of execution at the same time, and a use case for this is when you want to run two independent activities that perform different processes to an input from the same node. Next is the wait state. The wait state allows you to add a delay to your State Machine before it continues on executing states. This is similar to how a sleep function works. Now let's talk about the fail state. This state stops the execution of the State Machine and marks it as a failure. One example where you could use this is when you have a registration process and you want to deny any invalid form that a user has submitted. You can work the registration as a failure and have a custom error message like Invalid Username. The Succeed state, on the other hand, will just stop the execution successfully. And next is the passive state. You can use the passive state to pass the input from the previous state to its output without performing work. This is useful when you're debugging a portion of your State Machine, and you want to have a visibility of the output of a particular state. Maybe the data is formatted incorrectly and it is sending that down to another state, which makes your State Machine fail. The last state is the map state, and this state takes in an array as an input. You can think of the map state as a four-loop condition where you are basically iterating over each item in the array, and you're executing the same function to different parameters. So why use Step Functions, one use case where you can easily understand why Step Functions could be useful is when you have a sequential job implemented in a Lambda function. Imagine there are a lot of utility functions here that are chained sequentially, so whatever the output of function A is taken as an input for function B. The output of function B acts as an input to function C and so on. The problem here is that if something unexpected happens while, say, function C is running, which causes the Lambda function's execution to fail, you would have to invoke the Lambda function again to initiate the workflow. Since AWS Lambda is stateless, whatever output that function B has already produced will be lost. So basically you're back to zero again. One thing you could do is break these functions into separate states in a State Machine using Step Functions, so we have a task state for each function A, function B, and function C. AWS Step Functions has built in retry and error condition handling capabilities that you can implement for each state. So when the time function C fails, you would only need to retry the execution of function C. The state from function B is retained. Step Functions can also help solve the problem of timeout errors for a Lambda function. Let's say that there are four functions in your Lambda function and each of those functions runs sequentially for five minutes, we will get a total of 20 minutes if we add older execution times. This is a problem since Lambda can only run for a maximum of 15 minutes. To solve this, we concern the functions inside the Lambda function, into individual states, this way, each function is contained in a separate Lambda function, which has a separate execution timeout.

When you create an Amazon VPC, you must choose a primary CIDR block for your network. A CIDR block simply allows you to specify how small or how big your network will be. The allowed block size for a VPC is between 16 to 28 netmask. Basically, a netmask tells you the total number of available hosts for your network. The higher your network mask is, the fewer its available hosts will be. A 16-net mask translates to 65,536 IP addresses. If your cloud architecture will only use a handful of IP addresses, then you can use a smaller prefix. The smallest network that you can launch has a netmask of 28, which only has 16 IP addresses. Remember that AWS reserves a total of 5 IP addresses from your CIDR block. So, you have to consider that not all of the available IP addresses in your network will be usable. The first four IP addresses and the last IP address in each subnet CIDR block are reserved by AWS and are not available for you to use. You cannot assign these IP addresses to an instance. Say, for example, in a subnet with CIDR block prefix 24, the following five IP addresses are reserved. It starts from 10.0.0.0 all the way to.1,.2, and.3. The last IP address of the CIDR block, which is 10.0.0.255 is reserved for the network broadcast address. An Amazon VPC supports both IPv4 and IPv6. You can also create secondary CIDR blocks to further expand your VPC. It can operate in dual-stack mode, which means that your resources can communicate over IPv4 or IPv6 or both.
