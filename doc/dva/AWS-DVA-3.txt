AWS Serverless Application Model (AWS SAM) overview
- [Instructor] In this lecture, we'll be looking at AWS Serverless Application Model or SAM. AWS SAM is an open-source framework for building serverless applications. SAM is divided into two parts, the SAM templates and the SAM CLI. The first one that we're going to be talking about is SAM templates. A SAM template is actually an extension of a cloud formation template that lets you use shorthand syntax to define serverless resources, such as lambda functions, APIs, dynamo DB tables, and other event sources that trigger your lambda function. So how is it different from a cloud formation template? When writing a standard cloud formation template, we define resources for resource type. So if you want to build a backend API out of the lambda function in API gateway, you would need to declare the AWS lambda function resource. You must construct an execution role for that function, which is another resource type. And with API gateway, you have to do the same. You need to define the API resource, the API stage and so on. Now, the advantage of the SAM template is it provides you custom resource types, which allows you to write these resources in shorter lines of code. For example, AWS SAM has a resource type called AWS serverless function, which does a bit more than just creating a lambda function. This single resource type can build a lambda function, an IAM execution role, and an event source, and it's going to wire them altogether for you. All of these components can be created under a single resource type. Just imagine how much time and development effort you would save by using the SAM templates. To start writing a SAM template, you must declare the AWS serverless transform macro in the transform section of the cloud formation templates. The transform section lets you define one or more macros that will be used by AWS cloud formation to process your templates. In computing, a macro refers to a command that automatically expands to a set of simpler commands. In the case of SAM, the AWS serverless function resource is eventually expanded into several cloud formation resource types. When you deploy resources in a SAM template to cloud formation, cloud formation will first look for the transform section, and if there is one, and assuming that it is valid, all resources defined under that transform macro are transformed and expanded into native cloud formation resources. So at the end of the day, it's still interpreted as a normal cloud formation template. Take note that SAM has limited options in terms of its resource types. It's very specific to serverless resources. If you need to create resources that are not available in SAM like an EC2 instance, you can declare them in their native cloud formation syntax with no problem. AWS CodeDeploy in-place vs. blue-green deployment
- [Instructor] In CodeDeploy, you can either use an in-place or a blue/green deployment option for releasing code versions to your production environments. In this section, we will learn about their differences and their pros and cons so you can make a better decision when choosing a strategy for your deployment process. First, let's talk about how in-place deployment works. Say you have a web application running across EC2 instances inside a deployment group, and you want to update its user interface to make it more responsive. After developing the code for the new UI, the next thing you want to do is to archive the new application code along with an AppSpec file into a deployable content, which can be a ZIP or a TAR file. Take note that just like the buildspec, the YAML file, an AppSpec file must be located at the root level of your application folder. After building the archive, upload it to a repository that CodeDeploy supports. You can store the archive file to Amazon S3 and GitHub or Bitbucket. However, only EC2 and on-premises deployments can use GitHub and Bitbucket as repositories. When you deploy these new updates using in-place deployments, CodeDeploy will stop and uninstall the previous application version from your instances. What happens next is that CodeDeploy will pull the latest code version from the repository that you selected and install it to your EC2 instances. CodeDeploy will restart the application back into service once the deployment is completed. The disadvantage of an in-place deployment is that your application will experience an outage at the period of time during the deployment process. It also has a slower rollback compared to a blue-green deployment because it would take time to revert back the changes made to all instances. However, there is a way on how you can get around these disadvantages. And that is by modifying your deployment configuration. And by the way, what I just described here is an example of an all-at-once deployment configuration, which is also the default configuration for in-place deployments. We will discuss more about deployment configurations later on. For now, let's just focus on in-place and blue/green deployments. The biggest advantage of an in-place deployment is that it's cheaper to implement since you're only maintaining a single environment. Also, take note that in-place deployment is only available for deployments on EC2 and on-premises compute platform. Now let's go to blue/green deployments. In a blue/green deployment, you maintain two separate environments that have their own compute and network resources. Usually, each environment contains a load balancer that targets a group of computing resources. The blue environment refers to the active environment to which current user traffic is directed. The green environment is where the latest application revision is deployed. Once the new version has been successfully deployed to the green environment, the web traffic will be shifted away from the blue environment to the green environment, which now contains the updated version of your application. This cutover is done by using Route 53 to redirect the DNS records of the blue environment's load balancer to the green's load balancer. Once all user traffic is served by the green environment, the previous blue environment now becomes the green environment, and the current environment where the new version is installed becomes the new blue environment. The biggest advantage of a blue/green deployment is that switching to a new application version won't involve any downtime since you're just redirecting traffic, unlike in in-place deployments. The second advantage is that rolling back to the previous environment in case something goes wrong is much smoother because the switching's done at the network level. Also, since the blue and green environments are isolated from each other, you can perform tests on the green environments without affecting the operation of the blue environments. The obvious downside of a blue-green deployment is its expensive implementation cost that comes from running two environments at the same time.

AWS CodeDeploy deployment configuration
- [Instructor] Now let's look at the different deployment configurations available in CodeDeploy. What deployment configuration is used for is it allows you to define how aggressive or conservative your deployment approach is going to be. A deployment configuration is a set of rules and conditions that CodeDeploy applies during deployments to determine if a deployment is successful or not. There are five predefined deployment configurations in CodeDeploy. The first one that we'll talk about is the all at once deployments. All at once deployment is very easy to understand and it is the simplest among the deployment configurations. As the name implies, CodeDeploy will try to replace the old application code on all existing instances with the latest code version in one go. So if you have five instances, for example, the new application version is deployed to these instances all at the same time. So what are the requirements for a successful deployment? For deployment to succeed, the new application must be deployed to at least one instance. So even if the succeeding deployments fail, CodeDeploy will still mark that as a successful deployment. Only if all deployments to all instances fail does the overall deployment fail. This configuration isn't recommended for production workloads, however, its real value shines when you're deploying to test environments where you want to push changes and see them as quickly as possible. The second deployment configuration is one at a time. In this deployment configuration, the latest application version is rolled out to each instance one at a time. So if you have a fleet of instances, this could make your deployment process very slow. The overall deployment succeeds once the new application version is deployed to all of the instances. The exception to this rule is that even if deployment to the final instance fails, the overall deployment still succeeds. Say you have five instances in a deployment group. Even if CodeDeploy only succeeds in deploying to four instances, the overall deployment is considered successful. The deployment only fails if deployment of one of those first four instances fails. Next is half at a time. In this type of deployment configuration, your application is deployed to at least 50% of your instances at once. So if you have four instances, it will deploy to two of them simultaneously. If you have an odd number of instances, for example five, you take the half of five, which is 2.5, and round it down giving us a result of two. This means that your code will be deployed to two instances at once. However, for CodeDeploy to mark it successful, at least half of your instances must have a successful deployment. In the case of five instances, three is the next number larger than 2.5. But because we deployed two instances at a time, deployment will only succeed if the new application is deployed to at least four instances. We only take whole numbers into account because it's impossible to deploy to 0.5 of a computer. Okay, this deployment configuration is more conservative than the all at once deployments, and you won't have to wait as long as you would with one at a time. This is preferable if you're confident that at least half of your servers will be able to serve traffic without issue while the new application is being rolled out. The next one is Canary. These deployment configuration is only applicable to a blue-green deployment type. In a canary deployment, the shifting of traffic from the blue environment to the green environment is done in two increments. You can specify how much traffic will be shifted to your green environment at the first increment, as well as how many minutes CodeDeploy will wait before shifting the rest of the traffic. Let's take a look at this predefined configuration for a Lambda deployment. This says Lambda Canary 10%, five minutes. 10% means that at the first increments, 10% of user traffic will be shifted to the green environments. Five minutes means that the remaining 90% will be shifted immediately after five minutes. So by looking at the interval alone, which is five minutes, we can determine the total deployment time for this configuration. The last deployment configuration is Linear. like Canary, this deployment configuration is only applicable to a blue-green deployment type. In a linear configuration, traffic is shifted in equal increments. Let's take this predefined configuration for example. CodeDeploy default that Lambda linear 10% every one minutes. Just looking at how it is named, figuring what it does is pretty intuitive. So this configuration means that for every minute, 10% of the traffic will be shifted away from the old environments to the latest environments. The latest environment will serve 10% of total traffic in the first minutes. After two minutes, it'll serve 20% and it'll continue to shift 10% away from the old application to the latest application every minute until all user traffic is directed to the new application version. So in total, it would take about 10 minutes before all user traffic is shifted to the green environments. Take note that out of all these five, only all at once, canary and linear deployment configurations can be applied to AWS Lambda and ECS deployments. EC2 and on-premises deployments have no problem using any of these deployment configurations.

AWS CodeDeploy lifecycle event hooks
- [Instructor] The AppSpec file, as we talk about, is a piece of a configuration file that you need to include along with your application, particularly at the root level of your application folder. There are certain sections that must be defined when writing an AppSpec file, and some of those sections are dedicated to a specific compute platform, so we will not cover all of them. What we'll talk about in this lecture is the section that is common and the most important amongst all compute types, and that is the hooks section. The hooks section contains a set of logical groupings called lifecycle event hooks, which has their own set of instructions for deploying an application. Lifecycle event hooks are basically a list of scripts that CodeDeploy runs in sequence. Here's an overview of the lifecycle event hooks of an in-place deployment. Let's briefly go over each of them. The first lifecycle event is the Start event, and this initiates the instance for deployments. CodeDeploy automatically executes this for you, so you don't have to include this under your hooks section. What comes after the start event is ApplicationStop, and this event is used for running any scripts that will prepare your instance in receiving the new application version. For example, if you're deploying a new application version, say v.2, this event hook will allow you to include a script that will disable the previous version, v.1. Next is DownloadBundle. Like the Start event, this event gets executed automatically after the ApplicationStop event. As the name implies, during this stage, CodeDeploy will pull the new version, v.2, from a source repository into the instance. Next is the BeforeInstall event. The BeforeInstall event gives you the chance to do whatever you need to do before installing the new application version. A use case for this event hook is when you want to keep the previous configuration files of your old application, or maybe you want to back up some old logs and store them somewhere in your file system so they would not get overwritten when the new application comes. So after the BeforeInstall, the Install event will take place. During this event, CodeDeploy copies your new application files to the file destination that you specified. Again, this event happens automatically so you don't have to configure it on the hooks section. After the Install event comes the AfterInstall event. So usually this event is used for decrypting database string information, application secrets, or any encrypted data that you've sent down as a part of your application bundle and move them to the respected destination. Next is ApplicationStart. As the name implies, you use this event to restart your application back into service, which in this case is now v.2. Next is the ValidateService event, and you can use this event to run any validation script to check if the application is behaving as expected. Next is the End event, and this is the last lifecycle event hook. If all goes as expected, this event is where CodeDeploy will tell you that the deployment was indeed successful. If something wrong happens during the deployment, you'll have the chance to decide what steps to do next, like rolling back to a known working version for example. What's important to know here in the CD exam is the order of the lifecycle event hooks. This might seem a lot to take in at first, but if you really try to understand the function of each event, then you'll realize that you don't have to memorize all these lifecycle events because they're pretty intuitive anyway. For example, you can't immediately jump into the BeforeInstall event because, in an in-place deployment, the old application version must be stopped first to give way for the new version. You can't deploy a new version while the current version is still running. So even without having to memorize all these lifecycle events, if you understand the concept behind a deployment type, I'm pretty sure that you'll be able to answer questions regarding lifecycle event hooks.

AWS Elastic Beanstalk deployment policies
- [Instructor] Now let's talk about the different deployment policies that Elastic Beanstalk offers. Being a platform-as-a-service, Elastic Beanstalk does not only take care of the resource provisioning for your application, but can also help you simplify the process of deploying a new application version to your environment. A deployment policy simply defines how Elastic Beanstock carries out the deployment of your new code revision. The first deployment policy for Elastic Beanstock is the all-at-once deployments. As the name implies, when you deploy using the all-at-once deployments, Elastic Beanstock will deploy your new application code to all of your instances at the same time. So let's see how this works. Say you have an environment with autoscaled instances running behind a load balancer, and you deploy the new code version using all-at-once. What will happen next is that Elastic Beanstalk will temporarily disable your application across your instances and install the new code version simultaneously. When the deployment is successful, Elastic Beanstock will start the new application version back in the service. The clear advantage of an all-at-once deployment is its quick deployment time, since you are updating all instances at the same time. And this is something that comes in handy if you're doing tests in a development environment where you just want to release updates as fast as possible to see the results of your changes. Under all-at-once, your application will suffer a period of downtime that depends on how long your new code would take to be successfully deployed. That being said, this deployment isn't something that you would normally see in production and is only really helpful for doing iterative tests on developing environments. The second deployment policy is called rolling. So the idea behind a rolling deployment is that your new code version is rolled out in batches where you get to choose the batch size number. Let's see an example with a batch size of two instances. When you release a new code version to these environments using the rolling deployments, Elastic Beanstock will take two of your instances out of service at the time and install the latest code. So if your maximum capacity is six instances, anytime during the deployments, your maximum capacity will be reduced to four. It also means that some of your users will still see the old application version, and there will be some who will be interacting with the latest version. And this will happen until the deployment is completed. The advantage of this deployment is that the impact of failed deployment is lower compared to all-at-once. In our example, in case a deployment has failed, rolling back the changes will only apply to two instances. The rest of the instances are unaffected. Unlike all-at-once, this deployment policy does not have any downtime as long as the reduced capacity is able to serve the volume of traffic during deployments. The next deployment policy is rolling with an additional batch. So this is a variation of rolling, and it solves the problem of applications running below the maximum capacity during deployments. Let's see how this works. When you deploy the latest code version of your application to these environments, instead of taking an instance or instances out of service, Elastic Beanstock will provision a new batch of instances, install the latest code in them, and register the new instances to the load balancer. So in our example, Elastic Beanstock will create and deploy the code to two new instances. Take note that, during the deployment, your application is still using the old version and there is no reduction in capacity. After a successful deployment, Elastic Beanstock will register the two instances to the load balancer and will take out two old instances in return to go back to the defined capacity. This process will repeat until the latest application version is deployed to all instances. So what happens if the deployment fails? Well, if the first batch of instances fail, those instances are terminated, and your existing instances remain unchanged. But what if the first batch of instances succeeded, and the next batch after it has encountered a failure? In that case, both the latest and old versions of your code will be exposed to some of your users, so you need to redeploy the old version to revert back the changes that have been made. The advantage of this deployment policy is that your application will always run at full capacity. Since Elastic Beanstalk is spawning new instances on top of the existing instances, make sure that you don't exceed the number of EC2 limits for your account. For example, if you have a limit of 20 instances per region and you're already running 19 instances, if your batch size is two instances, Elastic Beanstalk won't be able to create those additional instances because it would violate your account limit, resulting in error. So be careful. The next deployment is called immutable. Immutable is synonymous with unchangeable, which implies that existing instances are not updated or modified in any way. In this deployment type, Elastic Beanstock will duplicate the existing instances into a fresh set of instances running in a separate Auto Scaling group. The new application version will be deployed to this new set of instances, and if the deployment is successful, Elastic Beanstock will register the new Auto Scaling group as a target for the load balancer and terminate the old Auto Scaling group. If something happens that has caused the deployment to fail, the new Auto Scaling Group will be terminated. The advantage of this is that rollbacks do not involve downtime and do not affect the existing instances where the working application version is running. Rollbacks are also faster because you're just terminating an Auto Scaling group. Your application will also run at full capacity during the deployments. This deployment strategy, however, is costly because you're duplicating the whole fleet of instances. Like rolling with an additional batch, this deployment is also subject to your EC2 account limit, so you need to be cautious if you plan to use this deployment method. The next deployment is traffic splitting, and this is great for A/B or Canary testing. A Canary test allows you to gradually release updates to a subset of your users while still serving your current application version to some of your users. In Elastic Beanstalk, you can set the percentage of traffic that is a split between the old and latest versions and the amount of time to monitor the new version before completing the deployment. When you deploy a new version using traffic splitting, Elastic Beanstock installs the new version to a new set of instances. It will then shift a percentage of traffic that you specified towards the new instances for a certain evaluation period. If the instances pass the health checks until then, the deployment is successful. Elastic Beanstock will route the rest of the traffic to the new instances and terminate the old instances. If the new instances did not pass the health checks or if you decide to cancel the deployments, Elastic Beanstock will simply reroute the specified percentage of traffic back to the old instances and terminate the new instances.

AWS Elastic Beanstalk lifecycle policy
- [Instructor] In this lecture, we will talk about how you can manage different versions of your Elastic Beanstalk application using the application version lifecycle settings. Whenever you deploy a new version of your application, whether via the console or the EB CLI, Elastic Beanstalk creates a new version of it with its own unique label. This is where the Elastic Beanstalk lifecycle policy comes into play. The lifecycle policy determines how old versions should stay before Elastic Beanstalk removes them from your account. Take note that the lifecycle policy only kicks in upon uploading a new version. That's why I said earlier that you should always maintain the total number of versions less than the application versions quota. So you can only choose either of these two. Next is the retention policy. Here you can select whether you'd like to retain the source bundle of your application version in Amazon S3 or not. So if you choose to retain the source bundle in S3, Elastic Beanstalk will still delete the application version from its record, but the source code of the version will still persist in Amazon S3. And if you choose to delete the source bundle in S3, then both the application version in Elastic Beanstalk and the source code in S3 will be deleted.

Amazon API Gateway overview
- [Instructor] Now let's look at Amazon API Gateway. Amazon API Gateway is an API management service that lets you create, publish, maintain, monitor, and secure APIs at any scale. So why would you use API Gateway in the first place? So imagine you have an application that talks to different microservices in your backend. In this case, each of the microservices owns a public endpoint that is used directly by your client application. This poses a flaw from a security standpoint as your microservices are exposed to anyone creating a large attack surface. Also, implementing security features like authorization and rate limiting requests to protect your backend from abuse would be quite challenging as you would need to apply them to each microservices. With API gateway, you can create a single point of entry that maps to your microservices. This provides an obstruction layer that hides all of your internal microservices from your clients making your application more secure. Routing requests to the proper microservices, enforcing authorization and rate limiting are all done on the API Gateway. API Gateway does not require any server management. So creating serverless architectures is a popular use case for it. Most of the time you'll see API Gateway coupled with AWS Lambda in serverless applications. Say you created an API, hosted it in Lambda function, and now you want other people to use it over the internet. How would you do that? You cannot directly attach a domain name to Lambda function. You can invoke Lambda function by its ARN, but that requires letting people access your AWS account. So here, API Gateway comes into play. With API Gateway you can expose your Lambda function through an API endpoint that can be distributed publicly. This way people can invoke your Lambda function without them having access to your accounts. You just have to give API Gateway enough permissions to invoke the Lambda function. You can also use API Gateway to modernize legacy applications. If you have a legacy application that you want to expose, say a SOAP web service that processes XML data, you can have clients send formatted in JSON and then have Amazon API Gateway transform that JSO data to XML using mapping templates. When you create an API, you get to choose between a REST API, HTTP API, and a WebSocket API endpoints. For most cases, the REST API is what you'll want to use. This gives you complete control over the request and response, along with other API management capabilities like caching, creating API keys and usage plans REST API also offers integration with other AWS services. HTTP API is cheap and is designed for simple low latency applications, but it lacks other API Gateway features and ask for the WebSocket API you may want to use this for applications requiring persistent connections such as real-time chat applications. It is also important to remember in the exam that API Gateway does not support unencrypted endpoints. When you deploy an API, by default, you get unencrypted endpoints. When you create a REST API, you get to choose an endpoint type. A private API endpoint allows a client to securely access private API resources inside a VPC. Amazon API Gateway overview
- [Instructor] Now let's look at Amazon API Gateway. Amazon API Gateway is an API management service that lets you create, publish, maintain, monitor, and secure APIs at any scale. So why would you use API Gateway in the first place? So imagine you have an application that talks to different microservices in your backend. In this case, each of the microservices owns a public endpoint that is used directly by your client application. This poses a flaw from a security standpoint as your microservices are exposed to anyone creating a large attack surface. Also, implementing security features like authorization and rate limiting requests to protect your backend from abuse would be quite challenging as you would need to apply them to each microservices. With API gateway, you can create a single point of entry that maps to your microservices. This provides an obstruction layer that hides all of your internal microservices from your clients making your application more secure. Routing requests to the proper microservices, enforcing authorization and rate limiting are all done on the API Gateway. API Gateway does not require any server management. So creating serverless architectures is a popular use case for it. Most of the time you'll see API Gateway coupled with AWS Lambda in serverless applications. Say you created an API, hosted it in Lambda function, and now you want other people to use it over the internet. How would you do that? You cannot directly attach a domain name to Lambda function. You can invoke Lambda function by its ARN, but that requires letting people access your AWS account. So here, API Gateway comes into play. With API Gateway you can expose your Lambda function through an API endpoint that can be distributed publicly. This way people can invoke your Lambda function without them having access to your accounts. You just have to give API Gateway enough permissions to invoke the Lambda function. You can also use API Gateway to modernize legacy applications. If you have a legacy application that you want to expose, say a SOAP web service that processes XML data, you can have clients send formatted in JSON and then have Amazon API Gateway transform that JSO data to XML using mapping templates. You can then use API Gateway to trigger rest service that posts the XML as a payload to the SOAP web service. When you create an API, you get to choose between a REST API, HTTP API, and a WebSocket API endpoints. For most cases, the REST API is what you'll want to use. This gives you complete control over the request and response, along with other API management capabilities like caching, creating API keys and usage plans REST API also offers integration with other AWS services. HTTP API is cheap and is designed for simple low latency applications, but it lacks other API Gateway features and ask for the WebSocket API you may want to use this for applications requiring persistent connections such as real-time chat applications. It is also important to remember in the exam that API Gateway does not support unencrypted endpoints. When you deploy an API, by default, you get unencrypted endpoints. For HTTP and REST APIs you get an HTTPS endpoint. And for WebSocket APIs, you get a WSS endpoint, which is similar to how an HTTP endpoint is encrypted with DLS. When you create a REST API, you get to choose an endpoint type. When you use the Edge optimized API endpoint, your API is deployed to the specified region while using a CloudFront distribution to facilitate client access typically from across AWS regions, API requests are routed to the nearest CloudFront point of presence. Regional API endpoint, on the other hand, routes an API request to the region specific API Gateway without going through any CloudFront distribution. A private API endpoint allows a client to securely access private API resources inside a VPC. Private APIs are isolated from the public internet and they can only be accessed using VPC endpoints for API Gateway that have been granted access.

AWS CodePipeline overview
- [Instructor] Now, let's talk about AWS CodePipeline. So far, we've seen AWS CodeCommit as version control, AWS CodeBuild as a continuous integration service, and CodeDeploy as a continuous deployment service. With these three services, you can implement the popular strategy in software development known as CI/CD. The CI in CI/CD, as you may already know, stands for continuous integration, and CD may refer to continuous deployment or continuous delivery. You might encounter different variations of CI/CD, but at the end of the day, the ultimate goal of it is to give organizations the ability to release code revisions as safely and quickly as possible out to production. We've seen what continuous deployment is, but we're not familiar with continuous delivery yet. AWS CodePipeline is actually a continuous delivery service, and we'll get to it in a while. What I want to talk about first is the difference between continuous deployment and continuous delivery. These two could be confusing as they both aim to do the same thing, and that is to deliver codes out to production. Here is a diagram of a CI/CD implementation with continuous deployments. Imagine this is you as a developer. Once you push a code update to a version control service, which is CodeCommit in this case, it will trigger CodeBuild, which creates and runs a build server where your code undergoes a set of tests to ensure you're only deploying clean code. If the test is successful, CodeDeploy will automate the deployment of the code revision to the production environment. The idea with this approach is that all of the stages inside this pipeline do all the work automatically from the version control up to the production environment. So someone writes a code, pushes it to a repository, then the rest is taken care of by the CI/CD pipeline. So there's no human intervention. If you trust your test enough and have full confidence in them, then this might be a reasonable approach for you. However, not everyone likes the idea of fully automating the deployment process from start to end, and sometimes a human must take over, review and make that final approval or decision before finalizing everything. And this is essentially what continuous delivery is all about. The key difference between continuous delivery and continuous deployments is the presence of manual approval. In continuous deployments, every revision is deployed to production automatically without explicit approval. With continuous delivery, not everything is done automatically. Every change that is committed triggers an automated workflow that builds, tests, and deploys the update to a staging environment where you can review the change. The updated version of the application is not pushed to production unless someone gives the green lights. So the final decision to approve or reject the deployment to production is up to you or someone that has the authority to do so. AWS CodePipeline is a fully managed continuous delivery service that can help you easily build your own CI/CD pipeline. A pipeline represents a series of steps that are sequentially executed to deliver code changes to production. You can organize your own stages in CodePipeline. A stage is simply a component that represents a step in your pipeline. A pipeline should contain at least two stages and only the first stage can have source actions, which makes sense because the first stage is what triggers your whole pipeline. A simple pipeline workflow might compose of just a source stage and deploy stage. The build stage is optional. The source stage contains a source action that is triggered whenever you push a code update to a repository. Then the deploy stage is where the update is finally deployed to your production server. You can modify the setup by adding a staging stage and a manual approval action stage before proceeding to the final deployment stage. At the point of the manual approval action stage, CodePipeline will pause the deployment so that anyone with enough permissions can perform necessary tests such as regression testing to check for possible bugs caused by the code change. And based on the outcomes of his or her evaluations, he or she can reject or accept the action. If the manual approval action is accepted, CodePipeline will resume the deployment to production. Otherwise, the deployment will fail.
