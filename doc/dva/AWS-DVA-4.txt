AWS X-Ray overview
- [Instructor] Now let's talk about AWS X-Ray. AWS X-Ray is a distributed tracing service that can help you analyze and debug potential issues in a distributed application. In a monolithic architecture, all processes that make up an application are tied to a single code base. This means that these processes share the compute in memory capacity of the server where the application is running. The problem with this setup is you cannot scale a specific function or process in your application. If you want to add capacity, you need to scale the entire application. In the case of a microservices architecture, your application is divided into smaller services that run on separate servers. One of the advantages of this setup is that each service may now scale independently, if necessary. Integration of new features is also easier because you don't have to deal with the entire application. Because these services communicate over APIs, one of the challenges of implementing a microservices architecture is the added network latency between each service. So aside from monitoring performance metrics and application logs like you do in traditional applications, you now also have to deal with latency monitoring to make sure that your application is always running in optimal condition. In a real world scenario, a microservices application consists of many more components. Since each service has its own access logs and performance requirements, monitoring and debugging your application as a whole becomes more complicated. This is where AWS X-Ray comes in handy. With AWS X-Ray, you can easily pinpoint which part of your application is causing an error or performance bottleneck. AWS X-Ray gives you visibility on the user request as they travel through the components of your application via the service map feature. The service map provides a visual representation of the connections between the services of your application. At the service level, we can see information about the average latency in failure rates for all services. The color scheme is also a great help because it lets you quickly spot the possible point of errors. So for example, with just a quick glance of the service map, you can already tell which nodes are causing the error or slowing down your application. AWS X-Ray also supports end-to-end tracing. And this is helpful if you want to trace the path an individual request has taken across your application nodes. You can use AWS X-Ray on applications running on Amazon EC2, Amazon Elastic Container Service, Elastic Beanstalk, or even on AWS Lambda.

Concurrency in AWS Lambda
- [Instructor] AWS Lambda is inherently scalable, but like many other things, there's a limit to how much it can scale, and it is important that you become aware of this, especially when you're building a high throughput application using Lambda functions. Concurrency is the number of Lambda functions that can run simultaneously at a given period of time. It is what determines how much Lambda function can scale. AWS imposes a default limit of 1000 unreserved concurrent executions across all functions for each region per account. However, you can overcome this limit and further increase concurrent executions up to hundreds of thousands by contacting AWS Support. You can reserve concurrencies across your functions, but the unreserved counts should not go below 100. These 100 units of concurrency are reserved by AWS for functions that do not have a reserve concurrency limit configured. This means that the maximum number of concurrent executions that you can reserve by default is restricted to 900 per region. Imagine we have three Lambda functions. Lambda function A has 150 reserve concurrent executions. Lambda function B has 250 reserve concurrent executions and Lambda function C does not have a concurrency limit set to it. Adding the concurrent executions of functions A and B yields a total reserve concurrency of 400. That leaves us with 600 unreserved concurrency executions. So what happens when the demand for Lambda function A exceeds 150 concurrent executions? This is where throttling comes into play. When a function hits its maximum concurrency limit, AWS Lambda will throttle the function to prevent it from serving incoming invocations. It is not possible for Lambda function A to tap into the remaining unreserved concurrency pool and borrow some of it. So you really need to be careful when planning the capacity for your Lambda functions. If you see that your function is being throttled too frequently, what you can do is increase the reserve number of concurrent executions for that function. Reserving concurrency to a Lambda function has the following advantages. First, it prevents your Lambda function from scaling out of control. If an event source suddenly sends an absurd amount of traffic to your Lambda function for whatever reason, your function will most likely consume all concurrencies in the region where your function is located. This is bad from a cost standpoint because you'd still have to pay for all of those invocations, and despite the fact that AWS Lambda is very cheap, ignoring this aspect of Lambda could still cost you a lot of money. The second advantage is it will prevent functions from blocking other functions from scaling. Without reserve concurrency, all of your Lambda functions within a region share the pool of unreserved concurrencies. If one function ends up eating a large portion of those concurrencies, other functions may not be able to scale up if necessary. This can be avoided by setting a concurrency limit to a function.

Amazon DynamoDB Global Secondary Index (GSI)
- [Instructor] Now let's talk about the two types of secondary indexes in DynamoDB. The first secondary index that we'll talk about is the Global Secondary Index, or simply GSI. Just like how it is named, the scope of queries that you can execute against a GSI is global. It means that a search can span all items in your base table, across all partitions. You can set up a global secondary index during the creation of a table or on an existing table. A DynamoDB table can have up to a default limit of 20 global secondary indexes. You can think of the GSI as a secondary table that can have a different partition key and sort key from that of your base table. So you can build a GSI that has a completely different schema design. Only projected attributes are accessible in a global secondary index. What this means is that if you try to query an item in a GSI whose original attributes are not included in the projected attribute, you won't be able to see that attribute associated with that item in the results. It is also important to note that a global secondary index has its own provision throughput for read and write operations. Queries and scans against a GSI consume capacity units from the index and not from the base table. So the read and write capacity of a GSI is separate from that of its base table. Bear in mind that applications do not write directly to the GSI. Any updates or deletes made to your DynamoDB table will be written first to the base table and reflected asynchronously to all of its GSIs using an eventual consistency model. So a single update made to the base table corresponds to two write activities, one for the base table and one for the GSI. Since a GSI has its own provision throughput, as a recommended practice, you should allocate a write capacity throughput that is equal to or greater than that of your base table to avoid potential throttling. In the next lecture, we'll have a demo on how to create a global secondary index.

Amazon DynamoDB capacity modes
- [Instructor] When you start using DynamoDB, you pay for the amount of data is stored on your table, the reads made against your items, as well as the write operations. There are optional DynamoDB features that you can use that incur additional charges, but if you're just using DynamoDB for what it is with its basic functionality, you pretty much only pay for the storage and capacity. In this lecture, we will look at the two capacity modes available in DynamoDB. The first one is the provisioned capacity mode. When you use this capacity mode, you have to set a fixed amount of read throughput and write throughput that will be allocated to your table. For provision mode tables, capacity throughput is specified in terms of capacity units. So for reads, we have the read capacity unit and write capacity unit for writes. We will just refer to them as RCU and WCU. The values that you can set for RCU and WCU can be totally different. These two are separate metrics, and there's no ratio between them of some kind that you need to maintain. If your table is expecting a read-heavy workload, then obviously, your RCU will be a lot larger than your WCU. If your table is more often used for writing items, then WCU will be bigger than RCU. The problem with the provision capacity mode is that your capacity is static. If you set five RCUs to a table and a burst of read requests suddenly comes in, overwhelming the capacity of your table, most of those requests will be throttled. To solve this problem, you can enable the DynamoDB Auto Scaling feature. Under the hood, this feature uses the AWS Application Auto Scaling service, which means it uses CloudWatch behind the scenes to track the metrics of interest for it to scale a resource. When you turn on Auto Scaling, you set the minimum and maximum capacity for that table. You can also tune the target utilization. The target utilization is basically the information that CloudWatch watches for so it will know when to scale in or scale out. With the Auto Scaling feature, you no longer have to settle for static or fixed capacity as it will dynamically add capacity up to the maximum level that you defined to serve requests. Of course, requests can still be throttled once the current capacity demand exceeds the maximum provision capacity. But nonetheless, this is a great feature to have as it gives your table a breathing space to adjust to the read and write demands of your application. The provision capacity mode is perfect for applications that have predictable or consistent traffic patterns. If this is the case, you can measure or estimate the required capacity for normal and peak workloads, and from there, you can calculate the minimum and maximum capacity for your table. This is also a great option for controlling costs, so if you're limited in budget and you want to have a specific monthly cost in mind that you're willing to pay, then go for the provision capacity mode. Next is the on-demand mode. When you choose the on-demand capacity mode, you're basically yelling DynamoDB to go wild and scale out as needed. This one doesn't need any capacity planning, so you don't need to specify how much read or write throughput your table needs. In on-demand mode, throughput is measured in read request units and write request units. DynamoDB will automatically adapt to your application's workload until it reaches the default request unit limit of your table. Just know that you can increase this limit upon request. The on-demand mode is good if you truly want a pay-per-use billing model. In the provision capacity mode, you always pay for the minimum provision capacity, regardless of whether your table is serving requests or not. The on-demand mode is suitable for tables that have unknown workloads or applications that have unpredictable traffic patterns. Since the capacity is restricted by the default table quota, which is by the way, in thousands, you should be careful as you could end up paying for more than what you are expecting.

Amazon DynamoDB RCU and WCU calculation
- [Instructor] Now let's talk about how you can calculate the required provision Read and Write Capacity Unit for your DynamoDB table. Let's start with the Read Capacity Unit. A Read Capacity Unit can support up to one strongly consistent read request per second for an item up to four kilobytes. This is also equivalent to two eventually consistent read request per second of up to four kilobyte items. For transactional read requests, it's going to cost you two RCUs to read an item up to four kilobytes in size. For reads on items greater than 4KB, the total number of reads required is equal to the total size divided by 4KB rounded up. Now let's go to the Write Capacity Units. It's easier to remember the rules for the Write Capacity Units. So a single Write Capacity Unit can support up to one standard write request per second for an item up to one kilobyte. As for transactional writes, you need two Write Capacity Units to perform one transactional write for items up to one kilobyte. For writes greater than 1KB, the total number of writes required is equal to the total item size divided by one kilobyte rounded up. All right, so let's have some examples to solidify your understanding of how to calculate the read and write capacity in DynamoDB. For our first example, let's find out the minimum read and write capacity of a five-kilobyte item that needs 10 eventually consistent reads per second and five standard writes per second. Let's first solve the RCU. Since the average item size is greater than 4KB, we will apply rule number three. So dividing five kilobytes by four kilobytes gives us 1.25 and rounding that up gives us two. This means that we need two RCUs to perform one strongly consistent read of a single five-kilobyte item. To get the eventually consistent reads, we will divide these by two and multiply it by the required reads per second, which is 10. If you are to solve for the RCU for 10 strongly consistent reads per second, you just need twice the number of eventually consistent reads, which makes it 20 RCUs, and it makes sense because a strongly consistent read is two times more expensive than eventually consistent reads. Now let's solve for the standard write. So again, one Write Capacity Unit is equivalent to one standard write request. To solve for the required Write Capacity Unit, we just have to multiply five kilobytes by the given writes per second, which gives us 25 Write Capacity Units. Let's have another example. This time we need to find the RCU and WCU of an eight-kilobyte item that needs 20 strongly consistent reads per second and two transactional writes per second. Since one strongly consistent read takes one RCU to read an item up to four kilobytes, we need two RCUs to read an eight-kilobyte item. Once we get to RCU, let's multiply the result by the given reads per second, which is 20. So the total RCU that we need to perform 20 strongly consistent reads per second is 40 RCUs. So we need to support two transactional writes per second. So this is easy. We just need to get the needed standard write, which we can directly solve for and just double the resulting value. So it takes eight Write Capacity Units to read an eight-kilobyte item. Double that value is 16 WCUs. Since we need two transactional writes per second, we will multiply 16 by two, which gives us 32 Write Capacity Units. And this is the answer that we're looking for.

Kung mayroon kang isang workload na nangangailangan ng paulit ulit na pagbabasa laban sa isang malaking hanay ng data, maaaring ubusin nito ang lahat ng RCU ng iyong talahanayan ng DynamoDB. Ito ay maaaring potensyal na makaapekto sa iba pang mahahalagang application na kailangang ma access ang iyong talahanayan. Kaya sa halip na basahin ang mga data na iyon mula sa talahanayan, maaari mong mahuli ang mga ito sa isang kumpol ng DAX. So kelan po ba dapat hindi DAX ang gamitin Dahil sa ipinamamahagi na likas na katangian ng isang kumpol ng DAX, maaari itong maglingkod sa malakas na pare pareho ang mga nababasa. Kaya kung mayroon kang isang application na hindi maaaring tiisin sa huli consistent reads, pagkatapos DAX ay hindi para sa iyo. Ang DAX ay hindi angkop para sa mga aplikasyon na mabigat sa pagsulat. Lalo na ito ay dinisenyo para sa mga workload na mabigat sa pagbabasa. Huling, maaaring gusto mong manibela ang layo mula sa DAX kung mayroon ka nang umiiral na caching layer, tulad ng Elastic Cache o Redis, at nais mong mapanatili ang iyong sariling lohika ng caching para sa iyong application.

Amazon DynamoDB Streams
- [Instructor] In this session, we're going to be looking at DynamoDB streams. DynamoDB Streams is a feature of Amazon DynamoDB that captures item level changes that occurs on a DynamoDB table. Once you enable it, anytime you perform a right operation, such as put, update or delete, DynamoDB will detect that as an event and send a modified record to a DynamoDB stream. The condition is that a change must be made to an item for it to be considered as an event. If you insert an item using an update or put operation, but do not actually change any data in that item, DynamoDB will ignore that and it won't create a stream record of that event. So in some way, it'll prevent duplicate records if ever a change is accidentally applied twice to one item. You can then set a Lambda function to process the records from the stream in batches. The records are set as an event object to your Lambda function, which causes it to be invoked synchronously. At the Lambda function, you can pretty much do anything you like with the records to suit your needs. For example, you can replicate the data to a separate data store like Elasticsearch for fast text searching purposes. Another example is if you want to aggregate data so you can use it for generating reports. Say you have a table that keeps transaction records and you want to have a collection of records with the total transaction amount for any given user ID. In that case, you can reformat the records using Lambda function and save them in a separate DynamoDB table that is dynamically updated whenever changes are made to the transactions table. When you enable DynamoDB Stream, you get to choose a stream view type. A stream view type determines what kind of data you are sending to a stream. So there are four stream view types that you can choose from. First is keys only. If you select this option only, the key attributes of the modified item will be written to the stream. So if you change any attribute of an item, only its partition key and sort key will be sent to the stream. Next is new image. As the name implies, DynamoDB will send the last updated version of the entire item to the stream, meaning not only the primary keys, but also including all the non-key attributes associated with that item. The third is old image. This time, instead of sending the modified version of the item, DynamoDB will send the entire item as it appeared prior to the update. And last is the new and old images. This is just a combination of the second and third stream view type. In this case, the old and updated version of the entire item will be written to the stream. As a result, you'll have two versions, the same item in the stream. Now let's see the key points that you have to remember in the exam. First, records is stored in a DynamoDB stream are removed after 24 hours, meaning you have to consume the stream records before they expire, otherwise you'll lose them. Next, DynamoDB writes it stream records in near real-time. In our example earlier, if you're pushing records to another DynamoDB table, then you can keep the target table in sync with the source table in near real-time without impacting the performance of the source table. Third, the stream records are ordered based on the sequence of item modifications in your DynamoDB table. And lastly, DynamoDB ensures that each record appears exactly once in the stream.

Amazon DynamoDB Time to Live (TTL)
- [Instructor] Now let's talk about a DynamoDB feature called TTL, or Time-To-Live. Time-To-Live allows you to automatically expire an item based on a defined timestamp. To set a TTL timestamp, first you have to enable TTL via the DynamoDB console or using the update Time-To-Live API. When you enable TTL, you get to assign the name of the TTL attribute that DynamoDB will look for in determining if an item is eligible for expiration. The name of the detail attribute is arbitrary. You can name it whatever you want, but it should be the same for all items that you want to expire at a certain period in time because you can only configure one TTL attribute per table. When adding a TTL attribute to an item, it should be of a number data type and its volume must be in Unix Epoch time format. Take note that TTL comes free of charge. You won't get billed every time an item is deleted and it doesn't consume any write capacity. So this is a very useful feature, especially if you're trying to keep your table size as small as possible without too much overhead on your part. Imagine, without this built-in TTL feature, you would have to build some kind of a custom chron job that scans your table in flushes items periodically. Apart from additional work, you would also have to pay for the read and write activities. TTL is a powerful tool for cleaning up your DynamoDB table to keep its sizes small and manageable over time, which would also translate into cost savings because you would have fewer data to read and store. TTL is also helpful for scenarios where you only need to keep data for a certain period of time, and after that, you need to get rid of them. For example, if you have an application that offers access to a service or product for a limited time, instead of writing the application logic to expire user access, it's a lot easier to just enable TTL and let DynamoDB handle it for you. A little caution though, when using TTL, TTL has a delay that increases proportionally to the table size. As a result, do not expect your items to be immediately deleted exactly after the timestamp that you set. At most, based on official documentation, DynamoDB deletes expired items within 48 hours of expiration, so you also have to factor that in if you're planning to use TTL in your application.

Amazon DynamoDB transactions
- [Instructor] Now, let's talk about DynamoDB Transactions. DynamoDB supports transactions that lets you perform ACID operations across one or more tables in an AWS region. ACID is a database property that stands for atomicity, consistency, isolation, and durability. Most of the relational databases adhere to the principle of ACID, and this is what makes them suitable for business-critical applications that deal with any kind of financial transactions, like banking, debit card processing, or order execution. So despite being a non-relational database, you get the same ACID support for DynamoDB with transactions. ACID assures the safety and integrity of a transaction. It ensures that there's no stale data, avoids data corruption, and offers overall accuracy and reliability of a database. I won't delve too much into each of the four properties. However, we'll briefly discuss atomicity so that we can better understand the implications and importance of ACID-compliant databases in scenarios such as financial transaction processing. Okay, so first, what is a transaction? A transaction is nothing but a collection of queries that effectively acts as a single unit of work. For example, let's say that you have two accounts in your bank. First, a checking account that you use for daily expenses, and second, a savings account where you keep most of your money. The checking account has $1,000 while the savings account has $10,000. Now, you want to transfer a portion of your savings account to your checking account. Say you want to deposit $1,000 off of your savings account to the checking account. So how do we carry that out in DynamoDB? First, we have to check the balance of the savings account to make sure it has sufficient amount to support the transaction. In DynamoDB, we can create a conditional check expression to verify if the balance has at least $1,000. If it has less than $1,000, then DynamoDB aborts the transaction. Otherwise, we proceed. Next, we subtract $1,000 from the savings account. To do this in DynamoDB, we have to use an UpdateItem operation. So we will update the balance attribute value of the savings account to $9,000. At this point, it's just a temporary change. DynamoDB won't commit the change and you won't actually see $9,000 in your savings account yet, because imagine if you did, you would freak out wondering where the $1,000 went, right? Remember, at the end of the day, the total balance of the two accounts should still be the same before and after the transaction, okay? Once DynamoDB has successfully updated the savings accounts, it'll proceed to the third step. To complete the transfer, the last thing left to do is to add the $1,000 to the checking accounts. For that to happen, we'll have to use another UpdateItem operation to change the balance attribute value of the checking account to $2,000. Then, DynamoDB will finally commit the changes. If you check your balance in both accounts, the sum of their balances should still be equal to their sum before the transfer. Now, going back to atomicity, the atomicity property requires the transaction as a whole to be successfully executed. Or if a part of the transaction fails, then the entire transaction is invalidated. So just like an atom, which is the smallest unit of matter, it's indivisible. An atomic transaction is indivisible in the sense that all actions within a transaction operate as a single unit. So basically, it's either all operations succeed, or if something bad happens along the way, it's better to just fail all the operations. It's all or nothing. In our example, the condition for the transaction to succeed is if all three operations also succeed. If somehow, the second operation has failed, DynamoDB will roll back any changes and fail the whole transaction. Same thing with the first or third operation. If one of these fails, then the whole transaction fails. So this is essentially what DynamoDB Transactions is all about. It allows you to group operations into a single unit. DynamoDB provides two transaction operations. These are TransactWriteItems and TransactGetItems. With the TransactWriteItems, you can group up to 25 write actions in a single all-or-nothing operation. The actions that you can add are the PutItem, UpdateItem, DeleteItem, and ConditionCheck operations. The TransactGetItems contains a read set with one or more GetItem operations. Each transaction can include up to 25 unique Get items or up to four megabytes of data, including conditions. Transactions are two times more expensive than Standard reads or writes because DynamoDB needs to perform two underlying reads or writes of every item in the transaction; one to prepare the transaction and one to commit the transaction.

Amazon DynamoDB global tables
- [Instructor] Now let's look at DynamoDB Global Tables. DynamoDB Global Tables allows you to automatically replicate your DynamoDB table across AWS regions of your choice. If you're building towards a multi-region architecture for an application, the advantage of this for you as a developer is you no longer have to supervise the replication of data yourself. DynamoDB manages the synchronization of tables via DynamoDB Streams. So before you can enable global tables, you have to enable DynamoDB Streams with the NEW_AND_OLD_IMAGES StreamViewType. So how does it work? Say, for example, you have global tables located in the N. Virginia region, Cape Town region, and another one in the Singapore region. If a user writes a new item in the table in N. Virginia, that change will be propagated a synchronously across the tables in Cape Town and Singapore. The same logic is true if a user writes to the table in Cape Town, the tables in N. Virginia and Singapore will have the exact copy of that new item. Now here's the tricky part. What if multiple users update the same item in different regions at near the same time? Suppose you have an application that uses a subjects table and this table is replicated across all three regions. Say that a user in Singapore changes the schedule of Math107 from 8:00 AM to 10:00 AM to 10:00 AM to 12:00 PM. And another user from Cape Town changes the assigned instructor code from 521 to 302 at almost the same time with only a split second difference. Now there will be conflict because changes are being made to the same item at once. So the way DynamoDB resolves this conflict is by using a last writer Wins reconciliation. This mechanism is done by comparing the timestamps of the updates. All tables will agree on whichever update has the most recent timestamp of the record. Since we have two modifications on the same item done at almost the same time, only one of those modifications will be accepted and replicated across all tables, whichever is the latest. In our case, it's either the update on the schedule or the instructor code. So with this approach, you might run into a situation where the instructor code and the schedule are not representative of what it really is in real life, hurting the reliability of your application. While there are some applications that can tolerate these small bumps in data accuracy, there are applications like in our example, in which accuracy might be vital to the operation because it could mess up the schedules of other classes and things like that. With that said, you need to put this last writer wins condition into consideration when building an application with global tables. Another thing that you should be aware of when using global tables is that the redirection of requests to tables in other regions in case of failures is not done automatically by DynamoDB. So what does this mean? As a developer, you still need to implement the logic that handles the redirection of traffic to other tables in case Amazon DynamoDB goes down in a particular region. Okay, so why would you use DynamoDB global tables? First, you get to protect your application from regional failures, and while regional failures might seem like a rare occurrence, you can't be too complacent about it. It's always a great move to anticipate it and prepare against that kind of scenario. There's a famous quote by Werner Vogels, the CTO of Amazon, which says "Everything fails all the time." So it's just a matter of when it'll happen. DynamoDB Global Tables helps you mitigate the impact of AWS regions being down. If one region experiences some issues that negatively affect your application, you can always shift the traffic to a working region. Second, if you have a latency sensitive application with globally dispersed users, obviously, you'd have to keep the latency as low as possible no matter where your users are located. Latency is directly correlated to distance. So for example, say you have a DynamoDB table in Singapore, users in the Philippines that have access to the DynamoDB table in Singapore will have a relatively better user experience than users in the United States because they are geographically nearer to their data. Hence, the latency will be lower. The further your target users are from your backend server, the higher the latency or delay they will experience. And although we're just talking about delays in the millisecond level here, in certain types of workloads, even an extra 50 millisecond delay could be the difference between what's good and an acceptable. So what you can do is you can replicate your table in the N. Virginia region, Ohio, or any AWS region that is closest to your target users.
